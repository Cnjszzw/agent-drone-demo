# WVP Copilot 异常场景与容错设计

> Agent 项目真正的区分度不在"调了什么 API、用了什么框架"，而在异常场景的处理策略。
> 本文档列出 14 个异常场景，每个场景说明问题、影响、解决方案和面试话术。

---

## 一、LLM 层的不可靠性

### 1. LLM 输出格式不稳定

**问题**：LLM 返回的 tool_call 参数可能包含非数字字符串（`"height": "100m"`）、缺失必填字段、或 JSON 格式错误。

**影响**：工具函数参数校验失败 → 整个任务链中断。

**已实现**：
- `_clean_tool_args` 函数：正则去掉参数中的单位后缀（`m`、`秒`、`x`），强制转换为数值类型
- LangChain `handle_parsing_errors=True`：LLM 输出格式异常时自动将错误回传 LLM 重试一次
- Pydantic 参数校验：`@tool` 装饰器自动根据类型注解生成参数校验，类型不匹配直接拒绝

**进一步设计**：
- 如果 `_clean_tool_args` 处理后仍有无法转换为数值的参数（如 `lat: "上海"`），返回明确错误而非静默降级
- 增加参数合理性校验层：`lat` 在 18-54、`lng` 在 73-135、`height` 在 10-120
- 关键参数缺失时，不调用 LLM 重试（LLM 不知道正确的坐标），而是返回提示让用户补充

**面试话术**：
> "LLM 的输出质量不能用'信任'来保证。我们在工具函数的入口做了三层校验——格式清洗（去掉单位后缀）、类型校验（Pydantic 自动拦截类型错误）、业务合理性校验（SafetyGate 验证坐标和高度范围）。这三层都不经过 LLM，是确定性的代码逻辑。框架层面的 JSON 解析错误由 handle_parsing_errors 自动重试。如果三层都过了，这个指令才是可信的。"


### 2. LLM 选择错误工具

**问题**：用户说"往前飞 500 米"，LLM 可能调用 `take_photo(count=500)`。

**影响**：意图执行错误——用户想飞，结果拍了 500 张照。

**解决方案**：
- System Prompt 中明确各工具的适用场景，但这是软约束
- 更可靠的方案：在 Plan 生成后增加一步"plan 校验"——规则引擎检查步骤序列的合理性：
  - `take_photo` 连续出现 3 次以上 → 可能误判
  - 飞行类指令（`fly_to_point`）的 count 参数不可能是 `500` → 人工确认
  - 工具调用链中同时出现 `return_home` 和后续操作步骤 → 业务逻辑冲突
- 校验失败不直接拒绝，而是标记 warning，前端展示时加警示颜色

**面试话术**：
> "LLM 偶尔会选错工具——比如把'往前飞 500 米'当成'拍 500 张照'。我们没有依赖 System Prompt 来避免这个问题。在规划生成后，加了一个轻量的规则引擎做 plan 校验：飞行类指令的参数范围不匹配、步骤序列逻辑冲突等，这些规则是硬编码的。校验失败不直接拒绝，而是标记 warning 让用户在前端确认卡片中看到。"


### 3. LLM 遗漏关键步骤

**问题**：用户说"飞到陆家嘴，拍照，返航"，LLM 生成的 plan 只有 `[maps_geo, fly_to_point, take_photo]`，漏了 `return_home`。

**影响**：飞机飞过去拍了照，不回。用户以为任务完成了，实际飞机还在目标点悬停耗电。

**解决方案**：
- 当前 System Prompt 中约束"return_home 必须是最后一步"——这是软约束
- Post-plan 校验规则（硬编码）：
  - 如果 plan 中包含 `fly_to_point` 但最后一个 step 不是 `return_home` → 自动追加 `return_home`
  - 如果 plan 中包含 `fly_to_point` 但第一步不是 `maps_geo`（且没有坐标） → 自动前置 `maps_geo`
- 设计原则跟 notify_frontend 一样——能硬编码修正的不交给 LLM 重试

**面试话术**：
> "LLM 偶尔会漏掉关键步骤，比如忘了返航。System Prompt 是软约束，不能依赖。我们在 post-plan 阶段加了规则校验——如果 plan 包含飞行但最后一步不是 return_home，自动追加。原则跟 notify_frontend 从 Tool 下沉到函数内部一样：能硬编码修正的，不需要 LLM 重新推理。"


### 4. LLM 超时 / 限流 / API 不可用

**问题**：DeepSeek API 返回 429（限流）或 503（不可用），或网络超时。

**影响**：用户指令无法处理，整个 Agent 不可用。

**解决方案**：
- LangChain 层面配置 `timeout=60`，超时后自动抛异常
- 重试策略：指数退避（1s → 2s → 4s），最多 3 次重试
- 429 限流：读取 `Retry-After` 响应头，按提示时间等待后重试
- 最终失败降级：返回预设的降级话术——"AI 服务暂时不可用，请稍后重试或切换为手动操控模式"
- 关键：降级不意味着无人机失控。手动操控通道仍然可用。Agent 是辅助入口，不是唯一入口

**面试话术**：
> "LLM API 不可用是必然会发生的——但我们不能让它影响无人机的基本操控能力。Agent 层面做了 3 次指数退避重试 + 429 限流感知。如果最终仍然失败，降级为预设话术，提示用户切换鼠标手动操控。Agent 是辅助入口，不是唯一入口。急停和手动操控通道始终可用。"

## 二、工具执行层的失败传播

### 5. 步骤失败后的状态不一致

**问题**：
```
step1: fly_to_point → 成功（飞机在目标点）
step2: take_photo → 成功
step3: return_home → 失败（电量不足）
结果：飞机停在了目标点，Agent 任务链终止。下一个用户不知道这个状态。
```

**影响**：系统状态与用户认知不一致。下次发指令时，Agent 假设飞机在起飞点，实际在目标点。

**已实现**：
- 执行循环中每步失败后 `break` 中止后续步骤

**进一步设计**：
- 每个任务维护一个全局状态快照（Redis key: `agent:session:state`），记录：
  - 任务 ID
  - 当前步骤（已执行到第几步）
  - 执行结果（成功/失败/未执行）
  - 设备最终状态（位置、电量、飞行模式）
- 新会话启动时先读取状态快照，如果上次任务异常终止，Agent 提示："上次任务在第 3 步失败，飞机当前位置 XXX，电量 XX%。是否继续？"
- 这样每个用户、每个会话都知道无人机的真实状态

**面试话术**：
> "Agent 任务链中断后，系统状态和用户认知可能出现不一致。我们在 Redis 中为每个会话维护了任务状态快照——记录已执行到第几步、每步的结果、设备的最终位置和电量。新会话启动时自动读取快照，如果上次任务异常终止，Agent 会提示用户当前状态并询问是否继续。"


### 6. 工具函数本身超时——Agent 认为失败，物理世界仍在执行

**问题**：`fly_to_point` 轮询 Redis 等了 3 分钟，状态始终是 `in_progress`。可能是网络断了、无人机坠了、也可能只是 Redis key 没更新但飞机实际已经到了。

**影响**：Agent 宣布任务失败，但无人机可能还在飞，甚至已经到了。Agent 和物理世界的状态脱耦——这是最危险的 gap。

**已实现**：
- 轮询循环有 `max_wait` 超时机制（Demo 30s，生产 180s）

**进一步设计**：
- 超时不等于失败——超时只是"我们不知道状态了"。超时后的处理分两步：
  1. 最后一次查询 Redis：如果仍无结果，标记为 `unknown`
  2. 发送一条 MQTT 查询指令（ping 无人机），如果硬件回复说明通信正常只是 Redis 延迟
- 超时后向用户返回的不是"任务失败"，而是"任务状态未知——无人机可能仍在飞行中。建议查询设备状态后决定下一步"
- 后端异步启动一个"状态确认"协程，持续查询直到拿到结果，拿到后推送 WebSocket 通知前端

**面试话术**：
> "工具函数超时是最容易被忽视的。轮询超时不等同于任务失败——无人机可能已经到了，只是 Redis 更新延迟。超时后我们不直接宣布失败，而是标记为 unknown，向用户返回'状态未知——请查询设备状态后决定'，同时后台启动异步确认协程持续跟进。Agent 和物理世界的状态脱耦是无人机场景特有的难点——纯软件 Agent 不会遇到这个问题。"


### 7. 并发冲突——多用户抢同一设备

**问题**：用户 A 正在执行"飞到陆家嘴"，用户 B 同时发送"飞到机场"。同一个无人机，两个 Agent 会话。

**影响**：MQTT 指令互相覆盖，无人机在两个目标点之间摇摆。

**解决方案**：
- 这个机制在 WVP 的 DRC（Drone Remote Control）模式中已实现——操控前必须获取 Redis 分布式锁
- Agent 会话启动时同样需要获取 DRC 锁，获取失败则排队等待或返回"设备被 XXX 占用中"
- Agent 执行任务期间定期续约锁（心跳），如果 Agent 崩溃，锁自动过期释放
- 急停操作绕过锁检查——急停优先级高于一切

**面试话术**：
> "多用户并发控制是 WVP 已有的基础设施——DRC 模式下通过 Redis 分布式锁保证独占控制权。Agent 会话启动时也需要获取这个锁。Agent 崩溃了怎么办？锁有 TTL 过期机制，Agent 执行期间定期续约，崩溃后锁自动释放。急停绕过锁检查——安全优先级高于独占控制。"


### 8. 工具返回的错误字符串被 LLM 误解

**问题**：
```
fly_to_point 返回: "❌ GPS 信号丢失，无人机已触发自动悬停"
LLM 理解: "上一步失败了，那下一步应该返航吧"
→ LLM 调 return_home
→ 但无人机在悬停状态下需要先恢复控制才能返航——LLM 不知道这个业务逻辑
```

**影响**：LLM 在物理世界知识盲区做出危险的"合理推断"。

**解决方案**：
- 当前已实现：工具返回 `❌` 或 `⛔` 开头时立即 `break` 中止后续，LLM 没有机会继续推理
- 更进一步的防御：错误消息中附带"禁止后续操作"标记。如果 `break` 被绕过（极端情况），工具函数自身检查全局 `emergency_stop` 标志，拒绝执行
- 多层防御：System Prompt 约束 → break 中止 → 工具函数自检 → 硬件拒收

**面试话术**：
> "LLM 收到'GPS 丢失'的错误后会尝试'合理推断'下一步——比如返航。但无人机在紧急悬停状态下需要先恢复控制。这个问题不能依赖 LLM 的理解——我们在工具返回失败时直接 break 中止后续，LLM 没有机会继续推理。这是一个多层防御：System Prompt 约束、break 中止、工具函数自检、硬件拒收。每层都是独立防线。"

## 三、物理世界的不确定性

### 9. 急停后的状态恢复

**问题**：急停 → 无人机悬停 → 用户开新会话 → "返航"。但飞机当前位置不是起飞点，返航高度应该是从当前高度开始还是 100m？

**影响**：急停后状态不一致，新会话基于错误假设执行。

**解决方案**：
- 急停触发时，Redis 写入一个带时间戳的紧急停止事件：`agent:session:emergency_stop:timestamp`
- 新会话启动时读取该事件，如果存在且在有效期内（如 5 分钟内），Agent 自动执行"状态查询"获取当前实际位置
- 返航参数基于实时状态计算：`current_height` 而非默认 `100m`
- 急停事件过期后自动清除，恢复正常会话流程

**面试话术**：
> "急停后开新会话，上一个任务的状态不能丢掉。急停时 Redis 写入带时间戳的停止事件，新会话启动时读到这个事件就自动查询当前实际位置——返航高度基于实时高度计算，不是默认 100m。急停不是'关了就忘了'，而是'关了但记得为什么关的'。"


### 10. 电量预估误差 vs 实际消耗

**问题**：SafetyGate 估算耗电 17%，实际因逆风消耗 40%。到了目标点只剩 45%，不够返航。Agent 还在继续执行"拍照、返航"。

**影响**：实际电量不足以完成全部任务，但 Agent 不自知。

**解决方案**：
- 当前 SafetyGate 的 `_estimate_distance()` 是基于直线距离的简化计算
- 每个飞行步骤执行完成后，`fly_to_point` 工具返回结果中包含实际耗电（来自 OSD 遥测数据）
- 在 `fly_to_point` 返回后，Agent 执行下一步之前，加一个电量检查节点：
  - 如果剩余电量 < 返航所需 + 20% 余量 → 强制中断任务链，优先返航
  - 如果电量够返航但不够后续拍摄 → 标记后续步骤为"因电量跳过"
- 这个检查不交给 LLM——是硬编码的数学判断

**面试话术**：
> "SafetyGate 的预估值是直线的，实际飞行有逆风、绕飞。我们在每个飞行步骤完成后做实时的电量复算——基于 OSD 遥测的真实耗电数据。如果剩余电量不够返航加余量，强制中断后续任务优先返航。这个判断是硬编码的数学公式，不经过 LLM。"


### 11. 网络断连期间的指令积压

**问题**：MQTT 断连 30s，恢复后积压 5 条进度消息。Agent 轮询 Redis 拿到了最新状态，但"到达目标点"这条关键消息如果在断连期间被丢弃了怎么办？

**影响**：Agent 一直等到超时，实际飞机已经到达——任务延迟增加，电池白白消耗。

**解决方案**：
- MQTT 消息采用 QoS 1（至少一次），保证断连恢复后消息送达
- "到达目标点"这类关键状态变化，Java 侧做至少一次重试投递
- Agent 侧的 Redis 轮询本身就是兜底——MQTT 消息丢了没关系，每秒轮询最终能看到 `completed` 状态
- 这就是选择"Redis 轮询"而非"纯 MQTT 推送"的原因之一：轮询天然容忍消息丢失

**面试话术**：
> "网络断连时 MQTT 消息可能丢失——但这正是我们选 Redis 轮询而不是纯 MQTT 推送的原因。MQTT 消息丢了无所谓，Agent 每秒的 Redis GET 必然能看到最新状态。轮询在这里不是开销，是容错手段。QoS 1 重试 + Redis 轮询兜底，双重保证。"

## 四、系统级韧性

### 12. Agent 进程崩溃后任务恢复

**问题**：Agent 正在执行第 3/5 步，Python 进程 OOM 被 kill。重启后任务状态丢失，用户不知道发生了什么。

**影响**：任务孤岛——无人机可能在执行某个指令的中途，但控制端不知道。

**解决方案**：
- 任务状态不在 Agent 内存中，而在 Redis 中持久化：
  - `agent:task:{task_id}:plan` — 完整任务计划 JSON
  - `agent:task:{task_id}:progress` — 当前执行到第几步
  - `agent:task:{task_id}:status` — running / paused / completed / failed
- Agent 重启后读取 Redis 中的未完成任务，询问用户："上次任务'飞到陆家嘴'在第 3 步中断，是否继续？"
- 如果用户选择继续，从第 3 步恢复执行
- 如果用户选择放弃，执行清理逻辑（如发送返航指令）

**面试话术**：
> "Agent 进程崩溃是必然事件。我们不在 Agent 内存中维护任务状态——全部持久化在 Redis 里。重启后读取未完成的任务列表，提示用户'上次任务在第 X 步中断，是否继续'。这让 Agent 的服务恢复行为和普通后端服务一样——无状态的编排层 + 持久化的状态存储。"


### 13. SSE 断连后状态恢复

**问题**：用户正在前端看执行进度，SSE 连接断开。重连后不在当前步骤，看到的进度卡住了。

**影响**：用户体验差，不知道任务是否还在执行。

**解决方案**：
- 前端 SSE 断连后自动重连（EventSource 原生支持）
- 重连时携带 `Last-Event-ID` 请求头
- 后端收到重连请求时，从 Redis 读取当前任务进度快照，补推中间丢失的事件
- 补推逻辑：读取 `agent:task:{task_id}:progress` → 将已完成步骤的事件重放一次 → 继续实时推送
- 如果 SSE 重连失败超过 3 次，降级为前端轮询 `/api/agent/status`

**面试话术**：
> "SSE 断连后前端自动重连，重连时后端从 Redis 读取当前任务进度，补推中间丢失的事件。如果 SSE 重连反复失败，前端降级为每 2 秒轮询一次任务状态。不能让用户盯着一个卡住的进度条。"


### 14. max_iterations 触发的隐式死循环

**问题**：LLM 连续 10 轮都在调用工具（没有返回纯文本），AgentExecutor 强制停止。用户看到"执行超时"，但不知道哪个环节出了问题。

**影响**：用户一脸茫然，不知道发生了什么。

**解决方案**：
- `max_iterations=10` 触发后，不是简单返回错误，而是返回详细的诊断信息：
  - 每轮调用了哪个工具
  - 每轮 LLM 的中间输出（摘要）
  - 当前设备状态
- 诊断信息同时写入日志和返回给用户，方便排查
- 如果同一类型的指令连续触发 3 次以上同一工具，在第三轮之后终止并返回"检测到循环调用，已中止。请确认指令后重试"
- 循环检测在 AgentExecutor callback 中实现——独立的，不依赖 LLM 自觉

**面试话术**：
> "max_iterations 触发时，不是一句'超时'把用户打发了。我们返回详细的诊断信息——每轮调了什么工具、LLM 的中间输出、当前设备状态。同时在 callback 里加了循环检测——同一工具连续调用 3 次以上直接中止。线上排查 Agent 的诡异行为全靠这些诊断日志。"

## 补充：LangGraph 替代 AgentExecutor 的选型故事

> 面试高频问题："为什么用 LangGraph？和 AgentExecutor 有什么区别？"

**背景**：整个项目最初全部基于 LangChain AgentExecutor。线性任务链（飞到→拍照→返航）运行稳定。但环绕飞行上线后频繁出现问题。

**问题**：环绕飞行要求"每 30 度拍一张，满 360 度停止"——期望 12 张均匀覆盖。实际跑下来每次拍照数量 8-20 张不等，且经常多绕好几圈才停，电量白白消耗。

**定位**：AgentExecutor 的 ReAct 循环每次拍照都要调 LLM 判断"要不要拍下一张"。LLM 推理延迟 1-3s/次。无人机环绕速度 3m/s，3 秒飞出 9 米。30 度弧段（80m 半径）约 42 米——等 LLM 返回时，飞机已错过最佳拍摄点 3-9 米。这不是"不准"，是物理上不可行。

**解决**：将环绕飞行从 AgentExecutor 中抽离，用 LangGraph 建模子图。`orbit_loop` 节点内部每秒从 Redis OSD 轮询位置，atan2 计算角度，跨过 30 度整数倍触发拍照，满 360 度条件边退出。整个循环不经过 LLM。

**原则**：没有全换 LangGraph。主链路（线性任务）继续用 AgentExecutor。只有"不该由 LLM 决策的时序逻辑"用图替代。

**面试话术**：
> "一开始全用 AgentExecutor，线性任务没问题。但环绕飞行上线后发现拍照数量飘忽不定——排查根因不是代码，是 LLM 推理延迟。无人机 3m/s，LLM 一次推理 1-3 秒，飞机已经飞出去 3-9 米了。环绕飞行的'每 30 度拍照'是确定性的时序逻辑，不该由 LLM 决策。我们把这块抽出来用 LangGraph 子图：每秒轮询 OSD 算角度，代码决定拍照时机，条件的边决定退出。主链路保持 AgentExecutor——没有全换，哪块不该让 LLM 决策就换哪块。"
