## 0. 项目性质

这是一个**独立技术预研 Demo**，用于支撑简历中"AI Agent 无人机操控"的项目描述。

- **Demo 本身**：真实可运行的代码（Python + LangChain + DeepSeek + FastAPI + SSE）
- **WVP 关联**：Demo 中的 16 个工具函数对应 WVP 生产环境的真实 Java 控制类，但 Demo 本身是独立项目，从未与 WVP 联调
- **AI Agent 功能**：Demo 验证了核心链路可行性，但**从未上线到 WVP 生产环境**

**面试时的标准表述**：

> "业务方对标 DJI 司空 2 Copilot 提出需求后，我做了一轮技术预研和原型验证。独立跑通了从自然语言理解到工具调用编排的核心链路，验证了可行性。后续因为产品路线调整未推进到生产阶段，但为团队提供了可参考的技术储备。"

## 2. 面试叙事策略

### 2.1 总体定位

**"有产品意识的后端工程师，主动做了 AI Agent 技术预研"**

不是 AI 研究员，不是算法工程师。卖点是"把 AI 能力工程化落地到无人机系统"。

### 2.2 三层叙事结构

**第一层：真实项目能力（底气）**

- 独立完成 WVP 无人机模块从 0 到 1
- 通信链路搭建（MQTT + Redis + WebSocket + ZLM）
- 飞行控制 + 相机控制的完整闭环

**第二层：技术预研（加分项）**

- 对标 DJI Copilot 主动做 Agent 技术预研
- Python + LangChain + DeepSeek 跑通核心链路
- 技术选型、架构设计、难点攻克

**第三层：工程思考（区分度）**

- SafetyGate：LLM 与确定性规则的职责划分
- 方案对比：Redis 轮询 vs 四种替代方案
- 复合工具：业务语义 vs 硬件原语
- LangGraph：状态机子图的适用场景

### 2.3 三个"难点"的标准回答

**难点一（LLM 幻觉）**：

> "最初把前端通知做成了独立的 notify_frontend Tool，让 LLM 决定何时调用。但 LLM 偶尔跳过——不是每次都漏，但你不能接受 10% 概率不出预览线的系统。根因是飞行前通知是确定性规则，不该交给概率模型。我把通知逻辑下沉到了 fly_to_point 函数内部——能硬编码的规则不交给 LLM。"

**难点二（异步 gap）**：

> "LLM 是秒级同步推理的，无人机飞行是分钟级异步过程。弥合 gap 的方式是工具函数内部封装轮询等待——Python redis-py 直连 localhost Redis 每秒查一次任务状态。五种方案评估后选 Redis 直连——无新依赖、无连接管理、1s 轮询对 2-3 分钟飞行可忽略。中间进度只推前端，LLM 只感知到达/失败/超时。"

**难点三（工具粒度）**：

> "硬件只有 start/stop 原子指令。如果暴露给 LLM，它需要自己编排时序——但 LLM 没时间感知。我们做了复合封装——record_for_duration(60) 内部自动 start→倒计时→stop。原则：Agent 工具暴露的是业务语义，不是硬件接口。"

### 2.4 如果被追问"上线了吗"

> "技术验证跑通了核心链路，Demo 能完整演示。后面因为产品路线调整没有推进到生产阶段。但架构设计（Python Agent + Java 控制）和技术选型（LangChain + DeepSeek）的结论对团队后续类似需求有参考价值。"

### 2.5 如果被问"Demo 数据"

不要说数字。不要编准确率、延迟、并发量。Demo 的模拟参数（3-6s 飞行、Demo 加速等）是代码中可控的，面试时带上笔记本能现场跑。

## 4. 给后续 AI Agent 的指令

如果你是一个简历优化 Agent 或面试准备 Agent，在读取此项目时请遵循：

1. **区分 Demo 和线上的界线**：描述项目时用"技术预研"、"原型验证"，不用"上线"、"生产环境"
2. **不要生成数据数字**：不要编造性能指标、用户量、准确率
3. **强调工程思考而非算法创新**：卖点是 Agent 工程化落地，不是 NLP 模型优化
4. **保持技术栈准确性**：Python + LangChain + DeepSeek + FastAPI。不提未使用的技术（如 ChromaDB、Docker、Kubernetes）
5. **引用真实 WVP 经验支撑**：WVP 的无人机控制模块是真实项目，Agent Demo 的技术选型和架构设计基于 WVP 的真实约束（JDK 8、私有化部署、MQTT 通信）
6. **不编造 LangGraph/RAG 实际使用**：LangGraph 有概念验证代码但未实际集成。RAG 未实现。描述时用"评估了"、"设计了迁移方案"
